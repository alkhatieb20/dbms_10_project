\section{Application domain and motivation}
% ============================================================

The manual creation of exam papers and their corresponding solution keys is a time-consuming and error-prone process. Errors frequently occur in calculating the total score or in formatting. This project solves the problem through a database-backed question management system. Lecturers and teachers use the application to centrally store exam questions (including LaTeX code for mathematical formulas) and dynamically compile them into exam papers. 
A concrete usage scenario is a teacher creating a new question in the "Operating Systems" domain and adding it to the question bank. A second scenario is the automatic generation of an exam, where the system merges selected questions, properly formats them, and correctly aggregates the total score.

\section{Data model -- ER sketch}
% ============================================================

\begin{figure}[ht]
  \centering
  % Replace 'er-sketch.jpg' with your actual hand-drawn ER diagram file name
  \includegraphics[width=0.85\linewidth]{er-sketch.jpg}
  \caption{ER sketch of the Exam Database model: entity types, key attributes, relationships, and cardinalities (including an N:M relationship).}
  \label{fig:er}
\end{figure}

Figure~\ref{fig:er} shows the ER model of the system. The central entity types are \texttt{Courses}, \texttt{Topics}, \texttt{Questions}, and \texttt{Exams}. The key attributes are unique IDs (e.g., \texttt{course\_id}). A 1:N relationship exists between Courses and Topics, and between Topics and Questions. Particularly noteworthy is the N:M relationship between \texttt{Exams} and \texttt{Questions} (resolved by the junction table \texttt{Exam\_Questions}), as an exam consists of many questions, and a question can be reused across different exams. This relationship also stores the assigned points and the sequence order of the question in the specific exam.

% ============================================================
\section{From model to schema}
% ============================================================

The relational schema consists of five tables. Primary keys are \underline{underlined}, and foreign keys are in \emph{italics}:

\begin{itemize}
  \item \texttt{Courses}(\underline{course\_id}, course\_code, course\_name)
  \item \texttt{Topics}(\underline{topic\_id}, \emph{course\_id}, topic\_name)
  \item \texttt{Questions}(\underline{question\_id}, \emph{topic\_id}, question\_text, latex\_content, default\_points, difficulty)
  \item \texttt{Exams}(\underline{exam\_id}, \emph{course\_id}, exam\_title, exam\_date, total\_points)
  \item \texttt{Exam\_Questions}(\underline{\emph{exam\_id}, \emph{question\_id}}, sequence\_order, assigned\_points)
\end{itemize}

\textbf{Normalisation.} The schema is in the 3rd Normal Form (3NF) because all attributes are atomic, depend solely on their respective whole primary key, and there are no transitive dependencies between non-key attributes. This structure prevents data redundancy.

\section{API design}
% ============================================================

The backend provides REST endpoints via FastAPI. Read accesses are public, while all write accesses (creating, updating, deleting) are protected by an X-API-Key.

\renewcommand{\arraystretch}{1.2}
\begin{tabularx}{\linewidth}{@{}p{2cm}p{4.5cm}X@{}}
\toprule
\textbf{Method} & \textbf{Path} & \textbf{Purpose} \\
\midrule
\texttt{GET}  & \texttt{/courses} & Reads all courses and their topics. \\
\texttt{GET}  & \texttt{/exams/\{id\}/details} & Reads an exam including questions (JOIN via Exam\_Questions + aggregation of total points). \\
\texttt{POST} & \texttt{/questions} & Creates a new question (requires X-API-Key). \\
\texttt{POST} & \texttt{/exams/\{id\}/questions} & Assigns a question to an exam (requires X-API-Key). \\
\bottomrule
\end{tabularx}

\section{Operation with Docker Compose}
% ============================================================

The backend consists of two services in the \texttt{docker-compose.yml}: \texttt{postgres} (the database) and \texttt{api} (the FastAPI server). Sensitive credentials like database passwords and the X-API-Key are securely stored in a \texttt{.env} file. Both containers are deployed on the lecture server and communicate internally, while the API is exposed externally via HTTP.

\textbf{Risks.} One risk is that the points in the N:M relationship (table \texttt{Exam\_Questions}) might be incorrectly aggregated during updates. Another risk is that blocking HTTP requests (without threading or \texttt{after}-callbacks) could freeze the Tkinter GUI.

\section{Acceptance criteria and testing}
% ============================================================

\textbf{Acceptance criteria.}
\begin{itemize}[topsep=2pt]
  \item Write accesses without a valid X-API-Key in the HTTP header are rejected (API returns 401/403 Unauthorized).
  \item \texttt{GET /exams/\{id\}/details} returns the exact sum of points (JOIN + Aggregation) for the exam.
  \item When a topic is deleted, all related questions are removed (ON DELETE CASCADE).
\end{itemize}

\textbf{Test plan.}
\begin{itemize}[topsep=2pt]
  \item \textbf{Database:} Verification of PK/FK constraints, UNIQUE rules, and CASCADE deletions.
  \item \textbf{API:} Automated tests with \texttt{pytest} and FastAPI's \texttt{TestClient} (happy path and error cases like invalid X-API-Key).
  \item \textbf{Business rule:} A test case simulating an exam with at least three assigned questions, verifying whether the total points returned by the server mathematically match the sum, thus eliminating the risk of calculation errors in N:M relationships.
\end{itemize}
